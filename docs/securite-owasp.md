| A01 | Broken Access Control | Un utilisateur accède au monde/aux entités d'un autre | Autorisation vérifiée en couche service : toute lecture/écriture filtre sur `ownerId` (`world-service.ts`) ; les entités héritent de cette vérification **en cascade** (`entity-service.ts` réutilise `getWorld` avant tout accès) — un monde ou une fiche d'autrui rend la même erreur (404) qu'une ressource inexistante, aucune fuite d'existence. **Recherche (KAN-17)** : `searchEntities` suit le même patron — le `worldId` transmis par l'action vient du client (non fiable), `getWorld` le revalide avant toute requête ; une recherche ne peut jamais remonter une fiche d'un autre monde. **Mentions manuelles (KAN-22)** : les id d'entités mentionnées via `@` viennent du client (contenu de l'éditeur) — jamais de confiance directe : `reconcileManualMentions` revalide chaque id contre une vraie requête `where: { id: { in }, worldId }` avant toute écriture de `Relation`, un id d'un autre monde ne peut jamais créer de lien. **Ignorer/délier (KAN-23)** : `targetId` vient d'un formulaire client — `ignoreLink` revalide qu'il appartient bien au monde courant (`entity.findFirst({ where: { id: targetId, worldId } })`) avant tout upsert `LinkIgnore`/suppression de `Relation`, no-op silencieux sinon ; `listIgnoredTargets`/`ignoreLink`/`unignoreLink` utilisent `getEntity` (pas seulement `getWorld`) pour vérifier aussi que `entityId` appartient au monde, fermant un gap potentiel identifié dans les fonctions voisines (`listOutgoingLinks`, `listIncomingLinks`, `getIgnoredTargetIds`, `reconcileManualMentions`, `listWorldRelations` ne vérifient que l'appartenance du monde, pas celle de `entityId` au monde — mitigé en pratique par les points d'appel UI qui valident déjà `entityId` via `getEntity` en amont, mais non corrigé rétroactivement ici : hors périmètre chirurgical de ce ticket, signalé au dev référent). | `src/services/world-service.ts`, `src/services/entity-service.ts` ; tests dédiés (`world-service.test.ts`, `entity-service.test.ts`, cas « monde/fiche d'autrui » et « mauvais worldId ») ; vérifié en conditions réelles avec deux comptes (`TST-SEC-002`, `TST-SEC-003`). `src/services/relation-service.ts` (`reconcileManualMentions`, `ignoreLink`, `unignoreLink`, `listIgnoredTargets`) ; tests dédiés (« ignore silencieusement un id mentionné qui n'appartient pas au monde », « no-op silencieux si `targetId` hors monde ») |
